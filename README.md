# 🥫 Food-Safety Alert Classification — ML Capstone Project

Predicting the **hazard category** of EU food/feed safety notifications (RASFF) from the product, origin country, destination country, and date — an end-to-end, reproducible machine learning capstone project.

**🔗 Live dashboard:** _add your Streamlit URL here after deployment, e.g. `https://food-safety-alert-classification.streamlit.app`_

---

## 1. Business Problem & Use Case

- **Problem:** Given a food/feed product, its origin country, and destination country, predict which **category of hazard** (microbiological contamination, mycotoxins, heavy metals, foreign bodies, packaging/chemical migration, allergens, etc.) it is most likely to trigger, based on historical EU RASFF notifications.
- **Target users:** Food-safety regulators and border inspectors, importer/QA teams, and food-safety analysts.
- **Expected outcome:** A trained classifier plus an interactive dashboard that predicts the likely hazard category for a new product/origin/destination combination and visualises historical notification trends.
- **Business / social value:** Faster triage of incoming notifications, better-informed inspection prioritisation, and an accessible tool for smaller importers to understand historical risk patterns.
- **ML task:** Multiclass classification (text + categorical/date features).

## 2. Dataset

- **Source:** A public, English-language extract of ~31,600 real **RASFF (Rapid Alert System for Food and Feed)** notifications (`safety_reports_v1.csv.gz`), published in the companion repository for *Human-in-the-Loop Machine Learning* by Robert Munro: [`github.com/rmunro/food_safety`](https://github.com/rmunro/food_safety). RASFF is the European Commission's official food/feed safety alert system ([food.ec.europa.eu/food-safety/rasff_en](https://food.ec.europa.eu/food-safety/rasff_en)).
- **Collection method:** Downloaded directly via a single `pandas.read_csv()` call against the raw GitHub URL (fully reproducible, no manual steps, no scraping needed at run time).
- **Size:** ~31,600 raw records (>10,000 required), reduced modestly after cleaning and rare-class folding.
- **Fields:** reference ID, free-text report summary, case date, food/product, hazard, origin country, destination country.
- **Sensitive data:** none — this is aggregate, already-public regulatory data with no personally identifiable information.

## 3. Methodology

1. **Data cleaning:** drop duplicates, drop rows missing key fields, parse dates, normalise text case.
2. **Target construction:** the raw `hazard` field (~960 unique free-text values) is grouped into ~12 business-meaningful **hazard categories** via keyword rules (see notebook Step 4). Rare categories (<80 records) are folded away.
3. **Leakage prevention:** `report_text` and `hazard` are excluded from the feature set because the auto-generated report text literally names the hazard. Features used: `food`, `origin_country`, `destination_country`, `year`, `month` only.
4. **EDA:** descriptive statistics, top hazards/products/countries, notification trend over time, target class-imbalance chart.
5. **Feature engineering:** TF-IDF (1–2 grams) on the `food` text field; one-hot encoding of origin/destination (capped to top 15 + "Other"); year/month passthrough.
6. **Validation strategy:** 80:20 **stratified** train/test split + 3-fold stratified cross-validation during model selection. All preprocessing fit only on the training fold via an `sklearn` `Pipeline`/`ColumnTransformer`.
7. **Models tested (8):** Logistic Regression, Linear SVM, Complement Naive Bayes, Decision Tree, Random Forest, Gradient Boosting, XGBoost, LightGBM, K-Nearest Neighbours.
8. **Hyperparameter tuning:** `RandomizedSearchCV` (3-fold, macro-F1 scoring) on the top 2 baseline models.
9. **Evaluation:** accuracy, **macro-F1** (primary metric, due to class imbalance), classification report, confusion matrix, train-vs-test gap as an overfitting check.
10. **Explainability:** tree-based feature importance over TF-IDF terms and one-hot country flags.
11. **Responsible AI review:** bias from inspection-intensity variation across countries, fairness framing, and explicit "no guaranteed outcome" language (see notebook Step 11).

## 4. Results

Full, up-to-date metrics are generated by running the notebook and are saved to `outputs/model_comparison_full.csv` and `outputs/model_metadata.json`. In development runs, gradient-boosted tree models (XGBoost, then LightGBM/Gradient Boosting) clearly outperformed linear and instance-based models, with the final tuned model reaching **macro-F1 ≈ 0.8** and test accuracy ≈ 0.8 on the held-out set — see the notebook for the exact numbers from your run (results vary slightly run to run due to class folding thresholds and search randomness).

## 7. Limitations

- Predicts a hazard **category**, not certainty of an actual incident.
- Notification volume reflects inspection intensity as well as true hazard rates — a source of potential bias by origin country.
- Short, noisy product-name text limits precision on rare hazard categories.
- Trained on a historical snapshot; should be retrained periodically as new notifications arrive.

## 8. Future Improvements

- Add a structured product-category taxonomy instead of relying solely on free-text TF-IDF.
- Incorporate notification type (alert / information / border rejection) if sourced live from the RASFF portal.
- Explore transformer-based embeddings for the product-description field.
- Add SHAP-based per-prediction explanations to the dashboard.
